4、面向对象 ​

面向对象三大特性 ​

1、封装：private 隐藏对象内部属性，对外公开简单的接口，boolean 变量的 get 方法是用：is 开头

2、继承：单继承,默认继承 Object 类

3、多态

  • 动态联编：是程序的最终状态只有在执行过程中才被决定而非在编译期间就决定了，属性没有多态。(运行时多态)
  • 编译时多态主要指方法的重载
  • 多态三要素：继承、方法重写、父类引用指向子类对象(向上转型)

关键字 ​

1、static

  • 类变量仅有一份，共享，无this指针，类变量存在于方法区中
  • 静态方法只能访问静态的变量与方法，不能以任何方式引用this和super关键字（在调用该方法时，不会将对象的引用传递给它）
  • 静态代码块只执行一次，初始化类变量，是在类初始化时执行，不是在创建对象时执行。不能访问非static成员

2、final

  • 进化终止信号
  • 变量不能更改
  • 方法不能重写
  • 类不可继承

3、访问控制符

  • import java.util.*; //导入该包下所有的类。会降低编译速度，但不会降低运行速度。

  • 局部代码块：在方法中出现；限定变量生命周期，及早释放，提高内存利用率

抽象类与接口 ​

抽象类

  • 抽象类不一定有抽象方法，有抽象方法的类一定是抽象类或者是接口
  • 抽象类不能实例化，让子类继承，抽象方法仅有方法声明

接口（规范）

  • 接口不能实例化，但可以声明引用变量类型
  • 可以多继承、多实现
  • 多态：接口引用调用实现接口的类的方法
  • 公有的抽象方法 public abstract
  • 公有静态的常量 public static final
  • 编译器自动添加

接口与抽象类的区别？

  • 一个子类只能继承一个抽象类, 但能实现多个接口

  • 抽象类可以有构造方法, 接口没有构造方法

  • 抽象类可以有普通成员变量, 接口没有普通成员变量

  • 抽象类和接口都可有静态成员变量, 抽象类中静态成员变量访问类型任意，接口只能public static final(默认)

  • 抽象类可以没有抽象方法, 抽象类可以有普通方法；接口在JDK8之前都是抽象方法，在JDK8可以有default方法，在JDK9中允许有私有普通方法

  • 抽象类可以有静态方法；接口在JDK8之前不能有静态方法，在JDK8中可以有静态方法，且只能被接口类直接调用（不能被实现类的对象调用）

  • 抽象类中的方法可以是public、protected; 接口方法在JDK8之前只有public abstract，在JDK8可以有default方法，在JDK9中允许有private方法

//包装类的缓存--享元模式 静态且final的，避免重复的实例化和回收
private static class ByteCache {
    private ByteCache(){}
    //Byte,Short,Integer,Long类型的缓存范围是[-128,127],Character缓存[0,127]
    static final Byte cache[] = new Byte[-(-128) + 127 + 1];

    static {
        for(int i = 0; i < cache.length; i++)
            cache[i] = new Byte((byte)(i - 128));
    }
}

public static Byte valueOf(byte b) {
    final int offset = 128;
    return ByteCache.cache[(int)b + offset];
}

fail-safe

当集合的结构被改变的时候，fail-safe机制会在复制原集合的一份数据出来，然后在复制的那份数据遍历。就不会抛出异常,但存在以下缺点：

底层原理：

Head（头节点），StatelessOp（无状态操作），StatefulOp（有状态操作），都是继承自pipeline，可理解成pipeline的节点。每个Pipline中持有upStream（也叫previousStage）的引用，从而连成本pipelines链表。操作的串联，操作是通过匿名函数，封装在接口中，最后包装在Sink类中。每个pipeline节点都包含一个Sink，每个Sink包含一个Sink的域，持有下一个Pipline的Sink对象的引用。从而，Pipeline和Sink构成了宏观上的双向链表。

Sink包装成链是在终结操作调用之后通过Pipeline链，逆向遍历包装的，Sink接口定义begin()、end()、cancellationRequested()（是否可以结束操作，可以让短路操作尽早结束）、accept()四个方法

pipline节点和操作串成双向链表，在一次迭代中执行所有操作（标记）。

垃圾收集器psilon

A NoOp Garbage Collector JDK上对这个特性的描述是: 开发一个处理内存分配但不实现任何实际内存回收机制的GC, 一旦可用堆内存用完, JVM就会退出

用法 : -XX:+UnlockExperimentalVMOptions -XX:+UseEpsilonGC

选项-XX:+UseEpsilonGC, 程序很快就因为堆空间不足而退出

主要用途如下 :

1、性能测试(它可以帮助过滤掉GC引起的性能假象)

2、内存压力测试(例如,知道测试用例 应该分配不超过1GB的内存, 我们可以使用-Xmx1g –XX:+UseEpsilonGC, 如果程序有问题, 则程序会崩溃)

3、非常短的JOB任务(对象这种任务, 接受GC清理堆那都是浪费空间)

4、VM接口测试

5、Last-drop 延迟&吞吐改进

ZGC

ZGC 是一个可伸缩的、低延迟的垃圾收集器。 GC暂停时间不会超过10ms，既能处理几百兆的小堆, 也能处理几个T的大堆(OMG)，和G1相比, 应用吞吐能力不会下降超过15%，为未来的GC功能和利用colord指针以及Load barriers优化奠定基础，初始只支持64位系统（Experimental, 不建议用到生产环境。）

用法 : -XX:+UnlockExperimentalVMOptions –XX:+UseZGC
